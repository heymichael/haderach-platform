Platform Control Plane Foundation Requirements

This document defines the baseline requirements for the `haderach-platform` control plane, including docs shell behavior, generated documentation workflows, and deploy-surface consistency.

1. Objectives

  • Provide a stable control plane for hosting, routing, and docs surfaces under `haderach.ai`.
  • Ensure docs UX is consistent across platform and app repos via shared shell assets and contract.
  • Guarantee generated docs are reproducible and sync to served hosting output with clear verification steps.

2. In Scope

  • Platform docs shell at `/docs` with tabbed navigation and consistent styling.
  • Generated priorities and architecture docs (`.md` source -> `.html` served targets).
  • Requirements catalog + project pages under `/docs/requirements/*`.

3. Out of Scope

  • Application business logic implementation within this repo.
  • Deep app-specific behavioral testing owned by app repos.
  • Production auth/IAP enforcement details for future private docs hardening.

4. Functional Requirements

7. Acceptance Tests

IDTest ScenarioExpected Result
AT-01 Open `/docs/?tab=requirements` with populated catalog. Cards are displayed for each `projects[]` entry with title and summary.
AT-02 Click a requirements card. Selected project doc loads inline and back button returns to list.
AT-03 Run generation + sync scripts, then inspect `hosting/public/docs/` outputs. Generated/synced files are present and match source-side contract.
AT-04 Load docs pages and inspect headers/meta. Noindex directives remain in place.

8. Constraints and Assumptions

  • Constraint: Platform repo remains control-plane only; app business logic stays in app repos.
  • Constraint: Route contracts must remain stable for docs shell deep-link behavior.
  • Assumption: App repos can adopt canonical shell and source/deploy contract incrementally.
  • Assumption: Shared scripts remain lightweight and transparent for contributors.

9. Risks, Rollback, and Follow-Ups

  • Risk: Divergence between source and served docs if sync step is skipped.
  • Risk: Per-app variations could drift from shell contract if templates are not reused.
  • Rollback: Revert docs shell/template changes and resync `docs/` to hosting output.
  • Follow-up: Add requirements catalog generation helper if manual catalog edits become error-prone.
